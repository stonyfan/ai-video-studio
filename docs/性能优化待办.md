# 性能优化待办

## 1. analyze 并发化 + 进度日志 + 重试（已完成 2026-07-09）

### 背景

`vision_analyze.analyze()` 原串行调用 GLM-4V：387 场景 × ~30s/次 ≈ 3.2 小时，占流水线 80% 耗时。且无逐场景进度日志，UI 看起来像卡住。

### 实施

`vision_analyze.py` 的 `analyze()` 改造：
- **并发**：`ThreadPoolExecutor(max_workers=5)`，同一 API key 并行调用
- **进度日志**：每场景开始打 `[analyze] i/total sc=xxx`
- **重试**：失败重试 3 次，1/2 秒退避（兜底 429 限流/网络抖动）
- **线程安全落盘**：`threading.Lock` 保护 results dict + analyzed.json 写入
- **顺序稳定**：落盘按 scene id 排序；返回按原始 scenes 顺序（拍摄时序）
- **resume 保留**：启动读 analyzed.json 跳过已分析场景

### 改动文件

| 文件 | 改动 |
|---|---|
| `video_worker/vision_analyze.py` | `analyze()` 重写：并发 + 进度 + 重试 + 锁 |

### 预期收益

| 指标 | 改造前 | 改造后 |
|---|---|---|
| analyze 耗时（387 场景） | ~3 小时 | ~40 分钟 |
| 总耗时 | ~4 小时 | ~1.5 小时 |
| 进度可见性 | 两条 warning 间沉默 | 每场景一行 |

### 影响评估

- **成片效果：无影响**。每个场景 GLM 分析独立，并发只改调度顺序，输入相同输出相同
- **storyboard：无影响**。读取按 scene id 排序的 analyzed.json
- **API 成本：无变化**。总调用次数不变
- **风险**：并发过高触发限流 → max_workers=5 + 重试兜底

## 3. 去重重构 + target 凑准（已完成 2026-07-08）

### 背景

旧去重用 `(main_object, action_type)` 字符串精确匹配，问题：
- `main_object` 是单字符串自由文本，GLM 命名不一致 → 漏去重 / 误去重
- `target_duration` 字段写了但代码没用 → 出片时长不受用户控制
- 实测：敦煌 387 场景 → 去重剩 124 个 → 出片 122s（用户设 300s）

### 实施方案（A + C + E 组合）

**E. 多物体 + Jaccard 去重**：
- prompt 改 `main_object`（单字符串）→ `main_objects`（列表，1-3 个物体）
- 去重 key 从字符串精确匹配改成 Jaccard 相似度 ≥ 0.7
- 零依赖、零体积增量

**A. 不足时补回**：
- 去重后总时长 < target×0.95 时，从 dropped 场景按时序补回

**C. 超出时截断**：
- 去重后总时长 > target×1.05 时，按 cut_duration 降序保留 top N

### 改动文件

| 文件 | 改动 |
|---|---|
| `configs/prompts.yaml` | 4 个 triplet_detect 模板：`main_object` → `main_objects: ["<物体1>", "<物体2>"]`，加说明"列出 1-3 个主要物体" |
| `video_worker/validators.py` | `AnalyzedScene` 加 `main_objects: list[str]` 字段，保留 `main_object` 兼容旧数据 |
| `video_worker/vision_analyze.py` | `analyze_scene` 解析 `main_objects`，兼容 GLM 返回 list / 旧版字符串 / 顿号分隔 |
| `video_worker/storyboard.py` | 重写 `deduplicate`（Jaccard）+ `plan`（截断/补回） |

### 验证

- 单元 smoke test：Jaccard 计算正确，去重 + 补回逻辑通
- 旧数据兼容：GLM 返回旧版 `main_object: "壁画"` → 自动转 `main_objects: ["壁画"]`

### 后续可调

- Jaccard 阈值 0.7 可调（0.6 更激进去重，0.8 更保守）
- 截断策略当前按 cut_duration 降序，可改成按 best_moment 质量分（需 prompt 加质量评分字段）

## 4. 视频稳定（B + C 组合，已完成 2026-07-09）

### 背景

静物/特写拍摄时的意外抖动影响成片质量。当前系统既不识别也不修复抖动。需要智能识别 + 按需稳定。

### 设计原则

- **不能对所有片段稳定**：运镜（推拉摇移）是创作意图，稳定会破坏镜头语言
- **只对"该稳定却抖了"的片段处理**：静物/特写 + 实测有抖动
- 两个条件都满足才应用 vidstab

### B 方案：GLM 判断场景类型（该不该稳定）

改 prompt，让 GLM 返回 `needs_stabilization: bool`：

```json
{"best_frame": "...", "cut_duration": 1.0, "main_objects": [...],
 "action_type": "...",
 "needs_stabilization": true}
```

判断逻辑：
- 静物 / 特写 / 静态全景 → `true`（应该稳定）
- 运镜 / 跟拍 / 平移 / 推拉 → `false`（运镜是创作意图）

prompt 加说明：`needs_stabilization 判断该场景是否应为静止画面（静物/特写=true，运镜/跟拍=false）`

### C 方案：OpenCV 光流检测（有没有抖动）

preprocess 阶段对每个场景抽 N 帧（如 10 帧），算相邻帧光流位移：

```python
# 用 cv2.calcOpticalFlowFarneback 算相邻帧位移
# 平均位移 > 阈值（如 5 像素/帧）→ 标记为抖动
scene.shaky = mean_displacement > THRESHOLD
```

避免对本来就稳的片段白跑 vidstab（省时间 + 避免无谓裁剪）。

### 组合逻辑

```
if scene.needs_stabilization (B) and scene.shaky (C):
    apply vidstab in render
else:
    skip
```

### 改动文件

| 文件 | 改动 |
|---|---|
| `configs/prompts.yaml` | triplet_detect 加 `needs_stabilization` 字段 + 判断说明 |
| `video_worker/validators.py` | `AnalyzedScene` 加 `needs_stabilization: bool` 和 `shaky: bool` 字段 |
| `video_worker/vision_analyze.py` | 解析 `needs_stabilization`；新增 `detect_shaky()` 光流检测（放 analyze 阶段，非 preprocess） |
| `video_worker/render.py` | `cut_clip` 加 `stabilize`/`trf_dir` 参数；render 主函数按 `needs_stabilization and shaky` 启用 vidstab |

> 注：实际实施时光流检测放在 `vision_analyze.py`（analyze 阶段），而非文档初版设想的 `preprocess.py`。原因：analyze 已有 scene 上下文，且只在 `needs_stabilization=true` 时才跑光流，避免对运镜场景白跑。`preprocess.py` 未改动。

### vidstab 实现细节

render 阶段对需要稳定的 cut 分两步：

```bash
# 1. 检测（生成 trf 文件）
ffmpeg -i input.mp4 -vf vidstabdetect=result.trf -f null -

# 2. 应用稳定 + 裁剪补偿
ffmpeg -i input.mp4 -vf "vidstabtransform=smoothing=30:input=result.trf:zoom=1.1" \
  -c:v libx264 ... output.mp4
```

- `smoothing=30`：30 帧窗口平滑
- `zoom=1.1`：放大 10% 补偿边缘裁剪
- trf 文件可缓存到 `jobs/<id>/work/stab/<scene_id>.trf`

### 验证

- 手持抖动的静物素材：稳定后画面平稳，边缘无黑边
- 运镜素材：不应用稳定，保留原镜头语言
- 本来就稳的静物：不白跑 vidstab（节省时间）
- 日志输出：`[render] scene s123: stab applied (disp=8.2px > 5.0)` 或 `skipped (disp=2.1px)`

### 风险

| 风险 | 缓解 |
|---|---|
| 光流检测耗时（387 场景 × 10 帧） | 抽帧降采样到 5 帧；或只在 needs_stabilization=true 的场景跑 |
| vidstab 边缘裁剪导致分辨率下降 | zoom=1.1 补偿；超分可后期加 |
| GLM 误判运镜为静物 | C 方案光流兜底：即使 GLM 说该稳定，没抖动也不跑 |
| trf 文件体积 | 单场景 ~10KB，387 场景 ~4MB，可接受 |

### 工作量估计

约 1 天：prompt + validators + preprocess 光流检测 + render vidstab + 测试
