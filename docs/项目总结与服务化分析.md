# AI 视频智能剪辑服务 - 项目总结与服务化分析

> 基于"檀道2"项目的全流程复盘，从 37 段原始素材到 30 秒精华短视频的 AI 驱动剪辑流水线。

---

## 一、项目背景

**起点需求**：把 37 段 4K 手机视频（5GB+，调酒制作过程）剪辑成一个 30 秒竖屏精华短视频，加上 BGM。

**最终成果**：6 个版本迭代，从机械时序拼接演进到 AI 内容理解 + 高光瞬时编排。

## 四、关键技术点

### 视频处理

- **方向判断坑**：ffprobe 报告 `3840x2160` 但实际抽帧是 `2160x3840`（手机视频 rotation 元数据，ffmpeg 5+ 抽帧自动应用旋转）。**正确做法**：用 PIL 检查抽帧后实际 `(W, H)`
- **Windows ffmpeg 滤镜坑**：路径里反斜杠 `\` 和绝对路径冒号 `C:` 都会炸滤镜字符串。**正确做法**：用 `relative_to(ROOT).as_posix()` + subprocess `cwd=ROOT`
- **vidstab 两步法**：`vidstabdetect`（生成 trf）+ `vidstabtransform`（应用变换）
- **精确切割**：`-ss START -to END` 配合 re-encode（不要用 `-c copy`，关键帧对齐有问题）

### 重复检测（四层）

| 层级 | 方法 | 能抓到 |
|---|---|---|
| 字节级 | MD5（前 2MB） | 完全相同的副本 |
| 像素级 | 32×32 灰度 MD5 | 完全相同画面 |
| 感知级 | 64×64 aHash + 汉明距离 | 几乎相似画面 |
| **语义级** | **AI 视觉理解（mcp analyze_image）** | **同动作不同画面** ← 最终方案 |

### AI 视觉调用技巧

- **并发**：单消息 10-20 个工具调用，41 张图 5-10 分钟
- **Prompt 设计坑**：不能给数字示例（模型会抄）；要求严格 JSON 输出 + "先描述再给值"
- **三联图技巧**：把同一场景 25%/50%/75% 三帧横拼成一张图，AI 能感知动作节奏（早→中→晚变化），判断"最佳瞬间"比看单帧准

### 故事弧编排

- PySceneDetect 切的是**镜头边界**（内容切换点），不是**动作高潮**
- 一段 6 秒视频可能只有 0.5 秒是"酒液接触杯口"的高潮瞬间
- AI 决定 `best_frame`（哪帧最佳）+ `cut_duration`（切多长），用 `compute_cut()` 算精确切片位置

## 五、经验教训

| 教训 | 体现 |
|---|---|
| **不要相信元数据字符串** | ffprobe 报尺寸 ≠ 实际像素 |
| **AI 视觉"主体"会随上下文变** | 同一段视频，AI 在源画面和裁剪后画面里选的"主体"不同 |
| **AI 描述 ≠ 视觉相似** | 18_0/20_0 AI 标"真空/水浴"，视觉上都是"袋子+柠檬" |
| **机械排序 ≠ 故事逻辑** | 按文件名数字排 → 切菜→倒酒→切菜→倒酒 跳戏；按"鸡尾酒制作流程"排才连贯 |
| **去重不能只看 action 标签** | 同 action 不同对象可保留（量杯/壶/杯），同 description 才是真重复 |
| **30s 短视频 = 高光瞬时密度** | 1.2s/瞬时 × 25 个 > 整段 5-7s × 7 个 |

## 七、真正的差异化优势（护城河候选）

### ⭐ 优势 1：高光瞬时检测（三联图法）

**独特点**：业界主流是"场景边界切"，本项目是"场景内找最佳瞬时"
- 别人：1 段 6s 视频 = 1 个 6s 片段
- 本项目：1 段 6s 视频 = 三联图判断 → 切出最佳 1.2s

**竞品做不到的原因**：
- 主流 AI 剪辑工具（剪映、Capcut）用音频卡点 + 模板
- AI 视频模型（Sora、Runway）是生成不是剪辑
- 通用 Agent 框架（Manus、Coze）没这套 pipeline

### ⭐ 优势 2：四层重复检测

主流工具只做前 3 层（或前 1 层），**语义级重复检测是关键差异**——能识别"同一动作多次拍摄"。

### ⭐ 优势 3：内容驱动的故事弧编排

- 别人：按时序 / 文件名 / 音乐节拍
- 本项目：AI 看完所有场景后按内容逻辑排（如鸡尾酒制作流程）

## 八、和其他 Agent / SaaS 对比

| 产品 | 能否复刻 | 缺什么 |
|---|---|---|
| **剪映/Capcut**（字节）| ❌ 部分能 | 模板化，无 AI 内容理解，无语义去重 |
| **必剪**（B 站）| ❌ 部分能 | 同上 |
| **万兴喵影** | ❌ | 传统剪辑软件，无 AI |
| **腾讯智影** | ⚠️ 部分 | 有 AI 但偏生成 |
| **Runway** | ❌ 不同赛道 | AI 生成而非剪辑 |
| **Pika/Sora/Kling** | ❌ 不同赛道 | 视频生成 |
| **Descript** | ⚠️ 文本驱动 | 播客剪辑，不适合视觉素材 |
| **Coze/Manus** 通用 agent | ✅ 能搭 | 但要重写整套 pipeline |
| **Dify** 自定义 workflow | ✅ 能搭 | 同上 |
| **阿里通义万相** | ⚠️ | 主生成 |

### 关键结论

**通用 Agent 平台（Coze、Dify、Manus）技术上能做到**，但他们没有：
1. 三联图法的独创算法
2. 四层去重的组合
3. 视频处理的工程化沉淀（Windows 滤镜坑、vidstab 参数、方向判断）

**所以护城河不在"能不能做"，而在"先做了 + 数据反馈迭代"**。

## 十、服务化的真正护城河建议

光有 pipeline 不够，护城河要靠这些：

### 🛡️ 护城河 1：垂直领域语料 + Prompt

通用 AI 看"调酒视频"只能给泛泛描述。如果**针对某个垂类**训练专属 prompt：
- 餐饮：识别"切菜-下锅-翻炒-出锅-装盘"故事弧
- 旅游：识别"出发-风景-特写-人物-结尾"
- 美妆：识别"卸妆-基础-化妆-成品"

**这就是无法被通用 agent 复刻的**。

### 🛡️ 护城河 2：用户偏好学习

让用户标记"我喜欢这种剪辑风格"，下次自动适配：
- 节奏快/慢
- 故事弧偏好（叙事 vs 卡点）
- 必保留元素（如必须有人脸）

### 🛡️ 护城河 3：素材库 + 模板库

积累"哪种 BGM 配什么故事弧"、"哪种转场适合哪类内容"，沉淀成知识库。

### 🛡️ 护城河 4：批量化工程能力

单条 30s 视频处理 5-10 元 AI 成本，但要做到：
- 1000 条/天 怎么并发？
- 不同分辨率/编码的源素材怎么自适应？
- 失败重试 + 进度追踪？

**这是工程壁垒**。

## 十二、其他 agent 能复刻吗？诚实回答

**能复刻的部分**（70%）：
- 视频合并、稳定、裁剪、加 BGM → 任何工程师都能写
- AI 视觉调用 → 接 Qwen-VL/Doubao API
- PySceneDetect → 开源工具

**不能直接复刻的部分**（30%）：
- 三联图判断最佳瞬时的算法
- 四层重复检测的组合策略
- 故事弧编排的 prompt 工程
- 大量踩坑经验（Windows 滤镜、rotation 元数据、ffmpeg 路径处理）

**结论**：技术 pipeline 3-6 个月可被复刻，**护城河要靠垂直 prompt + 用户数据 + 工程稳定性**。

---

## 十三、下一步建议

如果要做服务化：

1. **选一个垂类先打透**（如餐饮探店）—— 把 prompt 做到极致
2. **数据飞轮**：用户标记"满意/不满意" → 微调 prompt
3. **API 化**：把 pipeline 包成 HTTP API，前端做 Web/小程序
4. **降本**：把视觉调用切到 Doubao-vision-pro，成本降到原来的 1/5
5. **差异化宣传**：主推"AI 看懂内容"而不是"自动剪辑"
