# 候选片段提取流程：从原素材到 analyzed.json

## 概述

本文档说明 AI 视频剪辑管线中，**VLM 分析之前**的全部预处理逻辑——即原素材 MP4 是如何变成 `analyzed.json` 里 202 段候选片段的。

## 流程总览

```
源素材 *.MP4
    │
    ▼  [1] preprocess.py       重编码到统一规格
    │
    ▼  [2] scene_detect.py     内容检测 + 长段强切
    │
    ▼  [3] frame_extract.py    自适应抽帧序列
    │
    ▼  [4] vision_analyze.py   VLM 评分 + 元数据抽取
    │
    analyzed.json（候选池）
```

---

## [1] 预处理：`video_worker/preprocess.py`

### 作用
把任意规格的源 MP4 重编码到统一规格，方便后续场景切分、抽帧、渲染。

### 关键操作

| 操作 | 实现 |
|---|---|
| 方向检测 | 抽中间帧判断 h>w（竖屏）还是 w>h（横屏） |
| 旋转处理 | 依赖 ffmpeg 默认 `-autorotate`，把 displaymatrix metadata bake 进像素 |
| 横→竖转换 | 默认 `blur_background` 策略（高斯模糊背景填充，抖音/小红书主流） |
| 分辨率 | 720×1280（竖屏默认） |
| 帧率 | 25 fps |
| 编码 | libx264 / preset fast / CRF 18（视觉无损） |
| 音轨 | 去除（`-an`） |

### 三种横转竖策略

- `blur_background`（默认）：前景等比缩放居中，背景放大+高斯模糊填充
- `crop`：中央裁切（丢源画面较长一边）
- `letterbox`：上下黑边

### 输出
`jobs/<job>/work/clips/<stem>.mp4`

---

## [2] 场景切分：`video_worker/scene_detect.py`

### 作用
把每个 clip 按内容变化切成多个场景片段，生成后续 VLM 分析的最小单元。

### 核心算法：PySceneDetect ContentDetector

- **原理**：逐帧比较像素差异，差异超过 `threshold=27` 视为切点
- **最小长度**：0.4s（过滤抖动产生的微小片段）
- **兜底**：检测不到任何切点时，整段当 1 个场景

### 长场景强切（重要）

```python
if dur > max_scene_len_sec:  # 默认 12s
    n = max(2, math.ceil(dur / max_scene_len_sec))
    sub_dur = dur / n
    # 强制均分为 n 段
```

**为什么需要**：航拍/宣传片连续镜头常被 ContentDetector 视为单场景，导致素材池太稀疏。强制按时间窗口均分让后续 storyboard 有更多片段可选。

### 输出
`jobs/<job>/work/scenes.json`

```json
{
  "id": "<clip_stem>_<sc_idx>",  // e.g. C3962_0
  "src": "<clip_path>",
  "seg": "<clip_stem>",
  "sc_idx": 0,
  "start": 0.0,
  "end": 8.234,
  "dur": 8.234
}
```

## [3] 帧序列抽取：`video_worker/frame_extract.py`

### 作用
为每个场景抽 N 帧时序序列，作为 VLM 多图分析的输入（替代老的三联图方案）。

### 自适应帧数

```python
def _adaptive_frame_count(dur, min_n=5, max_n=12, factor=1.5):
    n = int(dur * factor)
    return max(min_n, min(max_n, n))
```

| 场景时长 | 抽帧数 |
|---|---|
| 1s | 5 帧 |
| 5s | 8 帧 |
| 8s+ | 12 帧（上限） |

### 抽帧规格

- **分辨率**：540×960（高于三联图单帧，因为不再横排拼接）
- **缩放模式**：cover（`scale + crop`，强制尺寸不变形）
- **时间点**：均匀分布在 `[start, end]`，端点包含
- **质量**：JPEG `-q:v 3`

### 输出
`jobs/<job>/work/frames_seq/<scene_id>_f00.jpg ... _fNN.jpg`

### 为什么不用三联图

Phase 13 改造：12 帧序列替代三联图。原因：
- glm-4v-plus 上限 5 图，三联图占用太浪费
- 多帧序列能给 VLM 看完整时序，动作判断更准
- 单独保存（不拼接）便于后续 pHash 多帧对比

## [4] VLM 分析：`video_worker/vision_analyze.py`

### 作用
对每个场景的 N 帧序列调用 VLM，输出结构化元数据 + 主观评分。

### 调用配置

- **Provider**：Doubao-Seed-2.1-pro（`ep-20260712162006-kcfdm`）
- **模式**：多图（`analyze_images`，N≥2 时）
- **并发**：`ThreadPoolExecutor(max_workers=5)`
- **重试**：429/网络抖动重试 3 次（1/2/4s 退避）
- **resume**：读已有 `analyzed.json` 跳过已分析场景

### VLM 输出字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `best_frame` | str | left/mid/right，强校验 |
| `cut_duration` | float | 0.3-3.0 clamp，再守一道不超过场景时长 |
| `main_objects` | list[str] | 主要物体（兼容旧版字符串） |
| `action_type` | str | 动作类别（倒酒/摇酒/切水果...） |
| `best_moment` | str | 高光瞬间描述 |
| `visual_quality` | int 1-10 | 画质主观评分 |
| `motion_score` | int 1-10 | 动态感主观评分 |
| `highlight_score` | int 1-10 | 高光主观评分 |
| `story_role` | str | opening/process/climax/ending/broll |
| `needs_stabilization` | bool | 是否需要稳像（静物=true，运镜=false） |
| `shaky` | bool | 实测抖动（仅 needs_stab=true 时跑光流检测） |
| `bad_reason` | str | 废片原因（空=正常） |

### 评分定义（configs/prompts.yaml 餐饮/调酒 prompt）

- **visual_quality**：9-10 极佳 / 5-6 中等 / 1-2 失焦
- **motion_score**：9-10 强动作 / 5-6 中等 / 1-2 完全静止
- **highlight_score**：9-10 出锅瞬间 / 5-6 中性 / 1-2 平淡

### 抖动检测（OpenCV Farneback 光流）

仅当 VLM 判断 `needs_stabilization=true` 时执行（节省算力）：
- 抽 8 帧
- 降采样到 720 宽
- 相邻帧 Farneback 光流，算平均位移
- `displacement > 5.0px` 视为抖动

### 输出
`jobs/<job>/work/analyzed.json`——**这就是 202 段候选片段的最终来源**。

## 关键设计决策

| # | 决策 | 理由 |
|---|---|---|
| 1 | 切分用 PySceneDetect 而非固定时间窗 | 内容差异比时间窗更接近"语义切分" |
| 2 | 长场景强切固定 12s 窗口 | 航拍/连续镜头兜底；理想是按内容变化分布切，但实现复杂 |
| 3 | 抽帧自适应（5-12 帧）而非固定 N 帧 | 短场景不需要那么多帧；长场景需要更多帧覆盖动作 |
| 4 | 用帧序列而非三联图 | 给 VLM 看完整时序，动作判断更准；同时绕开 glm-4v-plus 5 图上限 |
| 5 | 评分主观 1-10 整数 | 客观指标（直方图/光流）无法捕捉"高光感" |
| 6 | `needs_stabilization` 由 VLM 判断 | 运镜本身有位移，光流会误判为抖动；先让 VLM 分场景类型 |
| 7 | 重编码去音轨 | 后续会加 BGM，源音轨无用 |

---

## 已知偏差（VLM 不能完美判断语义的根源）

### 1. 切分按像素差异，不按语义
"倒酒"中途如果有手部晃动、镜头偏移，会被误切成 2 段。
**影响**：同一动作被分散到多个 scene_id。
**当前缓解**：timeline dedup 阶段合并相邻同 action_type 段。

### 2. 长场景强切是固定时间窗口
12s 一刀，可能把"摇酒 8s + 静止 4s"切成无关的两段。
**影响**：动作边界与切分边界不对齐。
**当前缓解**：暂无。

### 3. main_objects 是 VLM 文字描述，无标准化
同一物体可能用不同词："果盘+杏" vs "水果盘+黄杏"。
**影响**：语义合并难度上升，LLM 必须理解同义词。
**当前缓解**：timeline dedup prompt 明确告知"VLM 切片有偏差，相邻段动作一致就合并"。

### 4. 评分偏向动态画面
prompt 描述"9-10 出锅瞬间"，导致静态画面（白盘杏子、酒具特写）天然低分。
**影响**：纯按 highlight_score 过滤会丢故事性画面。
**当前缓解**：curation 流程改为"只砍 score≤2 明显废片"，故事完整性优先。

### 5. action_type 由 VLM 自由生成
没有白名单约束，可能出现"摇酒"/"摇晃酒杯"/"摇杯"等同义不同名。
**影响**：基于 action_type 的统计/过滤不准。
**当前缓解**：暂无（可选改进：加白名单校验或后处理归一化）。

## 改进方向（未来工作）

### 短期（不动主流程）
- **curation 阶段语义合并**：当前已经在 timeline dedup 实现，效果好
- **action_type 后处理归一化**：用 LLM 把自由文本归到固定 20-30 类

### 中期（动主流程）
- **scene_detect 加语义合并**：相邻场景如果 main_objects 同义，切完立即合并
- **长场景智能切分**：基于光流变化率找谷点，替代固定 12s 窗口
- **评分多模型投票**：visual_quality 这种主观指标，单模型容易偏，多模型平均更稳

### 长期（架构级）
- **音频前置**：ASR 转录 + 语音节奏切分，与视觉切分融合
- **镜头分组**：先识别"镜号"（同一机位/同一动作的多 take），再做场景切分
- **OpenCV 抽特征前置**：颜色直方图、光流、人脸检测等客观指标，作为 VLM 评分的校准输入
