# AI Video Studio 云 AI 剪辑方案

本文整理两种实现条件下，项目如何满足以下目标：

- 长视频：超过 30 分钟、容量超过 5GB，去除无用镜头，只保留高光片段。
- 多短视频：素材很多，从各视频中读取高质量内容并剪切，再按照故事弧拼接。
- 约束：主要依托智谱 API Key 对应的云 AI 能力，不依赖本地显卡。

## 当前项目状态判断

当前项目已经具备本地视频处理和云视觉分析基础：

- 本地扫描、预处理、场景切分、抽帧、渲染由 `video_worker/job.py` 串联。
- 当前智谱 GLM provider 主要使用 `chat.completions`，向模型发送 `image_url` 和 prompt。
- 当前抽帧逻辑是每个 scene 取 25%、50%、75% 三帧，合成三联图。
- 后端 vision proxy 当前只透传 `/chat/completions`。

也就是说，项目目前更接近：

```text
本地切分镜头
-> 每个镜头抽三联图
-> 云多模态模型做图像级评分
-> 本地规则 storyboard 排序
-> 本地 FFmpeg 渲染
```

距离目标产品还差一层：

```text
云 AI 理解代理视频片段
-> 输出候选高光和废片原因
-> 云 AI 生成全局剪辑决策表 EDL
-> 本地 FFmpeg 只负责执行剪切和渲染
```

参考：

- 智谱 GLM-4.6V 支持视频、图像、文本、文件输入，并支持 `video_url`：https://docs.bigmodel.cn/cn/guide/models/vlm/glm-4.6v
- 智谱 GLM-ASR-2512 支持音频转文本，单文件限制为大小不超过 25MB、时长不超过 30 秒：https://docs.bigmodel.cn/cn/guide/models/sound-and-video/glm-asr-2512
- 智谱 Embedding-3 价格为 0.5 元 / 百万 Tokens，支持 256-2048 维：https://docs.bigmodel.cn/cn/guide/models/embedding/embedding-3

## 方案一：只用一个智谱多模态模型接口

### 定位

这是最适合先做 MVP 的方案。

核心思想：

```text
本地只生成低清代理素材
-> 智谱多模态模型判断片段是否值得保留
-> 智谱多模态模型生成剪辑决策表
-> 本地 FFmpeg 按原片时间码执行剪辑
```

这里仍然只用一个多模态模型接口，例如 GLM-4.6V 的 `chat.completions`。区别是不能继续只发三联图，而要把输入从 `image_url` 扩展到 `video_url`，让模型分析低清代理视频片段。

### 长视频流程

```text
5GB 原视频
-> 本地生成低清代理片段，例如 30-60 秒一段
-> 上传代理片段，得到可访问 URL
-> 使用 video_url 发给 GLM 多模态模型
-> 模型输出每段结构化结果：
   keep / discard
   highlight_score
   visual_quality
   motion_score
   bad_reason
   recommended_start
   recommended_end
   story_role
   reason
-> 过滤废片，得到候选高光池
-> 再把候选片段元数据交给 GLM
-> 输出最终 EDL
-> 本地 FFmpeg 按 EDL 裁剪原片
```

建议切片策略：

- 粗筛阶段：30-60 秒一个代理片段。
- 精筛阶段：只对候选片段附近做 5-15 秒复查。
- 最终剪切：使用原始视频，不使用低清代理。

### 短视频多素材流程

```text
多段短视频
-> 每个视频生成低清代理或关键片段
-> GLM 多模态模型分析：
   画面质量
   主体
   动作
   情绪
   镜头价值
   是否重复/低价值
   适合的故事角色
-> 形成候选片段池
-> GLM 根据目标 prompt 生成故事弧：
   hook
   build_up
   process
   climax
   ending
-> 输出 EDL
-> 本地 FFmpeg 拼接
```

### EDL 输出格式建议

```json
{
  "target_duration": 90,
  "style": "story_arc",
  "clips": [
    {
      "source": "video_03.mp4",
      "source_start": 12.4,
      "source_end": 18.8,
      "target_role": "hook",
      "reason": "动作明确，画面冲击力强，适合作为开头",
      "transition": "cut"
    },
    {
      "source": "video_07.mp4",
      "source_start": 33.0,
      "source_end": 47.5,
      "target_role": "build_up",
      "reason": "补充事件背景，和开头形成连续叙事",
      "transition": "crossfade"
    }
  ]
}
```

### 能满足的效果

适合：

- 旅行
- 美食
- 探店
- 产品展示
- 运动
- 舞蹈
- 风景
- 视觉高光明显的视频

可以达到：

- 自动去除明显废片。
- 自动找画面质量较高的片段。
- 自动给镜头打高光分。
- 自动按故事弧拼接。
- 根据 prompt 修改剪辑目标。
- 不依赖本地显卡。

不稳定的场景：

- 访谈
- 直播
- 课程
- 播客
- 游戏解说
- 强依赖对白笑点、关键词、语义转折的视频

原因是只用多模态视频理解时，模型能理解画面和部分视频内容，但不如专门 ASR 稳定提取完整台词。

### 用户使用成本

成本结构：

```text
用户成本 = 多模态视频理解调用成本 + 最终 EDL 规划调用成本
```

主要成本来自多模态视频理解。最终 EDL 规划通常只处理候选片段摘要，文本量较小，成本相对低。

调用量示例：

| 场景 | 切片策略 | 预计多模态调用量 |
|---|---:|---:|
| 30 分钟长视频 | 60 秒一段 | 约 30 次 |
| 30 分钟长视频 | 30 秒一段 | 约 60 次 |
| 30 分钟长视频 | 10 秒一段 | 约 180 次 |
| 100 个短素材 | 每个素材 1 次粗评 | 约 100 次 |
| 100 个短素材 | 粗评 + 候选精评 | 约 120-150 次 |

降本原则：

- 不上传 5GB 原片，只上传低清代理片段。
- 第一轮只做粗筛，输出短 JSON。
- 只对 top 20%-30% 候选片段做精排。
- 用户改 prompt 时复用已分析结果，只重跑 EDL 规划。
- 同一个素材计算内容 hash，重复素材不重复收费。
- 将片段分析结果持久化到 job 缓存和素材库。

## 方案二：多模态视频理解 + ASR + 可能的 Embedding

### 定位

这是更接近成熟产品的方案。

核心思想：

```text
画面交给多模态视频理解
声音交给 ASR
素材检索和去重交给 embedding
最终由 GLM 统一生成 EDL
```

### 长视频流程

```text
原视频
-> 本地生成：
   低清代理视频片段
   音频切片
   关键帧
-> 多模态模型分析画面高光
-> ASR 提取台词、关键词、笑点、情绪转折
-> 合并画面分析 + 语音文本
-> GLM 生成候选高光池
-> GLM 全局规划最终 EDL
-> 本地 FFmpeg 精剪原片
```

### 短视频多素材流程

```text
每个素材
-> 多模态分析画面
-> ASR 分析语音
-> 生成片段摘要、标签、人物、动作、情绪、台词
-> embedding 入库
-> 根据目标 prompt 检索最相关素材
-> GLM 生成故事弧剪辑方案
-> FFmpeg 拼接
```

### 能满足的效果

相比方案一，方案二能明显增强：

- 对对白高光的识别。
- 对直播、课程、访谈、解说视频的适配。
- 对“笑点、冲突、金句、关键词”的发现。
- 多素材检索、去重、按主题选片。
- 用户二次改 prompt 时的响应速度和成本控制。

### 用户使用成本

成本结构：

```text
用户成本 =
多模态视频理解成本
+ ASR 成本
+ embedding 成本
+ 最终故事规划成本
```

其中：

- 多模态视频理解通常是主要成本。
- ASR 会随音频时长增加。
- Embedding-3 官方价格为 0.5 元 / 百万 Tokens，通常不是主要成本。
- 最终故事规划只处理摘要和候选清单，成本较低。

降本原则：

- ASR 只处理有人声或有声音变化的片段。
- 音频按 30 秒切片，满足接口限制。
- embedding 只对摘要、标签、ASR 重点文本入库，不对完整冗长文本反复入库。
- 多模态分析、ASR、embedding 都做缓存。
- 同一素材不同剪辑任务复用已有分析结果。

## 两个方案对比

| 维度 | 方案一：只用多模态接口 | 方案二：多模态 + ASR + Embedding |
|---|---|---|
| 上线速度 | 快 | 中等 |
| 工程复杂度 | 低到中 | 中到高 |
| 本地显卡依赖 | 无 | 无 |
| 长视频视觉高光 | 可用 | 更稳 |
| 对白类高光 | 弱 | 强 |
| 多素材语义检索 | 弱 | 强 |
| 去重能力 | 依赖 prompt 和规则 | 可用 embedding 增强 |
| 用户成本 | 较低 | 中等，但可控 |
| 产品成熟度 | MVP | 更接近正式产品 |

建议路线：

```text
先做方案一 MVP
-> 验证用户是否接受 AI 自动粗剪
-> 再接入 ASR
-> 最后加入 embedding 素材库
```

## 当前项目达到方案一还需要做什么

### 1. 从三联图分析升级为代理视频片段分析

当前问题：

- `frame_extract.py` 只生成三联图。
- `glm.py` 只发送 `image_url`。
- 这种方式只能做图像级评分，不能稳定理解动作过程、镜头节奏和片段完整性。

需要新增：

- 代理视频生成模块，例如 `proxy_clip.py`。
- 对每个 scene 或时间窗口生成低清短视频：
  - 360p 或 480p。
  - 低码率。
  - 去除无关高码率负担。
  - 保留音轨可选，但方案一可以先不依赖音轨。
- provider 支持 `video_url` 输入。

目标：

```text
AnalyzedScene 不再只来自 triplet image，而可以来自 proxy video clip。
```

### 2. 增加素材上传或临时可访问 URL 能力

当前问题：

- 智谱 `video_url` 需要模型可访问的视频 URL。
- 本地文件路径 `D:\xxx.mp4` 不能直接给云模型读取。

需要新增：

- 上传代理片段到可访问对象存储。
- 或后端提供临时文件服务和签名 URL。
- 上传后记录：
  - local_proxy_path
  - remote_url
  - expire_at
  - content_hash

目标：

```text
本地生成代理片段 -> 上传 -> 得到 video_url -> 调 GLM。
```

### 3. 新增 GLM video provider 调用

当前问题：

- `GLMProvider._raw_call()` 当前 content 只有 `image_url` + `text`。

需要新增：

- `analyze_video(video_url, prompt)`。
- 或将 provider 抽象升级为：

```python
analyze_asset(asset_type, asset_url, prompt)
```

需要支持的 content：

```json
[
  {
    "type": "video_url",
    "video_url": {
      "url": "https://example.com/proxy.mp4"
    }
  },
  {
    "type": "text",
    "text": "请分析这个片段..."
  }
]
```

目标：

```text
同一个智谱多模态接口既可分析 image_url，也可分析 video_url。
```

### 4. 重做 prompt：从镜头评分改成剪辑候选判断

当前 prompt 主要让模型返回单 scene 的评分字段。方案一需要它直接判断“是否保留”和“推荐剪点”。

建议输出：

```json
{
  "keep": true,
  "discard_score": 2,
  "highlight_score": 8,
  "visual_quality": 8,
  "motion_score": 7,
  "bad_reason": "",
  "summary": "人物走入画面并展示产品",
  "recommended_start_offset": 2.4,
  "recommended_end_offset": 8.9,
  "story_role": "hook",
  "tags": ["产品展示", "近景", "动作明确"],
  "reason": "主体清晰，动作有信息量，适合作为开头吸引点"
}
```

目标：

```text
模型输出可直接进入候选池，而不是只给 storyboard 一个分数。
```

### 5. 增加候选片段池 Candidate Pool

当前问题：

- `vision_analyze.py` 分析完直接交给 `storyboard.plan()`。
- 没有明确的候选池、废片池和精排池。

需要新增：

- `CandidateClip` 数据结构。
- `keep/discard` 分流。
- `discard_reason` 分类。
- `candidate_score` 统一评分。
- 每个候选片段保留模型理由，方便前端展示。

建议分类：

- keep：进入最终候选。
- maybe：二次精排。
- discard：废片，不进入 storyboard。

目标：

```text
先筛掉无用镜头，再规划成片。
```

### 6. 增加 EDL 全局规划步骤

当前问题：

- `storyboard.py` 主要是本地规则排序。
- 多素材故事弧需要全局模型判断，而不是只按分数排序。

需要新增：

```text
candidate_pool.json
-> GLM 全局规划
-> edit_decision_list.json
-> render.py 执行
```

全局规划 prompt 需要告诉模型：

- 目标时长。
- 平台风格。
- 用户 prompt。
- 片段候选列表。
- 每个片段的摘要、评分、标签、故事角色。
- 不要重复。
- 保持故事弧。
- 输出严格 JSON。

目标：

```text
从“规则 storyboard”升级为“AI 生成剪辑决策表”。
```

### 7. 渲染层支持 EDL

当前问题：

- `render.py` 当前消费的是 storyboard 结果。
- 方案一需要 render 能直接消费 EDL。

需要新增：

- EDL schema 校验。
- source/start/end/order/transition 字段支持。
- 推荐剪点越界修正。
- 片段最小时长和最大时长限制。
- 渲染后校验输出时长和文件有效性。

目标：

```text
模型只负责决策，本地 FFmpeg 按 EDL 稳定执行。
```

### 8. 成本控制和缓存

当前问题：

- 只要重新跑任务，就可能重复调用模型。
- 长视频和多素材场景会造成大量调用。

需要新增：

- 代理片段 hash。
- 分析结果缓存。
- 失败重试和断点续跑。
- 每个 job 的预计调用量和预计成本。
- 用户改 prompt 时只重跑 EDL，不重跑底层片段分析。

目标：

```text
同一素材只分析一次，多次剪辑复用结果。
```

### 9. 前端增加候选片段审核界面

当前产品如果只输出最终视频，用户很难信任 AI 的选择。

需要增加：

- 候选高光列表。
- 废片列表。
- 每段的 AI 理由。
- keep/maybe/discard 状态。
- 一键替换片段。
- 重新生成故事弧。

目标：

```text
不是让用户手剪，而是让用户快速确认 AI 的剪辑判断。
```

## 方案一最小可交付版本

建议 MVP 范围：

1. 本地生成 30 秒低清代理片段。
2. 上传代理片段并得到 `video_url`。
3. GLM 多模态分析每段，输出 keep/discard/highlight_score/recommended_start/recommended_end。
4. 过滤候选片段。
5. GLM 根据候选片段生成 EDL。
6. 本地 FFmpeg 用原片时间码裁剪。
7. 前端展示候选列表和最终视频。

暂不做：

- ASR。
- embedding。
- 精细音频节奏卡点。
- 自动字幕。
- 大规模素材库。
- 复杂贴纸和特效。

这版能验证的核心价值：

```text
用户导入长视频或大量短素材
-> 系统自动找候选高光
-> 系统自动给出故事弧剪辑
-> 用户只做少量确认
```

## 结论

如果目标是尽快验证“减轻人剪辑视频负担”，建议先做方案一。

方案一不是继续优化三联图，而是要做这几个关键升级：

```text
三联图 image_url
-> 低清代理视频 video_url
-> 候选片段池
-> AI 生成 EDL
-> FFmpeg 执行 EDL
-> 缓存和成本控制
```

等方案一能稳定筛出视觉高光后，再接入 ASR 和 embedding，覆盖对白型视频和多素材检索场景。
