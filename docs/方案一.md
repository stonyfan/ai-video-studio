# 方案一：当前项目达到 MVP 效果还需要做什么

本文只分析“只用一个智谱多模态模型接口”的方案一。

目标不是接入 ASR、embedding 或其他独立能力，而是在现有智谱多模态模型 API 基础上，把项目从“三联图评分 + 本地规则排序”升级为“代理视频理解 + 候选高光筛选 + AI 生成 EDL + 本地执行剪辑”。

## 方案一目标效果

方案一要满足两个核心场景：

- 长视频：超过 30 分钟、容量超过 5GB，自动去除无用镜头，只保留高光片段。
- 多短视频：素材很多，自动读取各视频中的高质量内容，再按照故事弧拼接。

方案一的合理产品定位是：

```text
AI 自动粗剪 + 用户快速审核 + 本地最终渲染
```

不是：

```text
一次性上传 5GB 原片，让模型直接生成成片
```

## 当前项目和方案一的差距

当前项目已有基础：

- 本地视频扫描、预处理、场景切分。
- 三联图抽帧。
- 智谱 GLM provider。
- 多 provider API key / proxy 基础。
- storyboard 编排。
- FFmpeg 渲染。

但当前链路仍然是：

```text
本地 scene_detect
-> frame_extract 生成三联图
-> GLM 分析 image_url
-> vision_analyze 得到单镜头评分
-> storyboard 本地规则排序
-> render 本地渲染
```

方案一需要变成：

```text
本地生成低清代理视频片段
-> 上传代理片段并得到 video_url
-> GLM 多模态视频理解
-> 生成候选高光池 / 废片池
-> GLM 全局生成 EDL
-> render 按 EDL 裁剪原片
```

核心差距有 7 个：

1. 当前输入太薄，只看三联图，缺少视频过程。
2. 当前没有代理视频片段生成。
3. 当前没有代理片段上传和 `video_url` 管理。
4. 当前 GLM provider 只发送 `image_url`，没有 `video_url` 分析方法。
5. 当前没有明确的候选片段池和废片池。
6. 当前 storyboard 主要是本地规则，不是云模型生成 EDL。
7. 当前缺少成本缓存和用户审核界面。

## 需要补的能力

### 1. 代理视频片段生成

目的：

```text
避免上传 5GB 原片，只上传低清、短时长、可分析的代理片段。
```

当前项目有三联图生成，但方案一需要新增代理视频生成模块。

建议新增模块：

```text
video_worker/proxy_clip.py
```

职责：

- 根据 scene 或固定时间窗口生成低清 mp4。
- 保留原始 source/start/end 映射。
- 控制代理片段大小。
- 生成 content hash，方便缓存。

建议参数：

```yaml
proxy_clip:
  enabled: true
  window_sec: 30
  resolution: "480p"
  video_bitrate: "500k"
  audio: false
  max_clip_mb: 20
```

长视频建议：

- 第一轮粗筛：30-60 秒一段。
- 第二轮精筛：只对候选片段附近切 5-15 秒。
- 最终渲染：仍然使用原片时间码。

验收标准：

- 30 分钟视频能生成约 30-60 个代理片段。
- 每个代理片段有 `source_path/start/end/proxy_path/hash`。
- 代理生成不依赖本地显卡。

### 2. 代理片段上传和 video_url 管理

目的：

```text
智谱 video_url 需要云端可访问 URL，本地 D:\xxx.mp4 不能直接给模型读取。
```

需要增加一个上传层。

可选实现：

- 上传到对象存储，例如 OSS/COS/S3。
- 或后端提供临时文件服务和签名 URL。
- MVP 阶段可以先用后端静态文件托管，但生产要加鉴权和过期时间。

建议新增数据结构：

```json
{
  "scene_id": "sc_0001",
  "source_path": "D:/videos/a.mp4",
  "source_start": 120.0,
  "source_end": 150.0,
  "proxy_path": "jobs/job_x/work/proxy/sc_0001.mp4",
  "remote_url": "https://example.com/proxy/sc_0001.mp4",
  "content_hash": "sha256...",
  "expire_at": "2026-07-13T00:00:00Z"
}
```

验收标准：

- GLM 可以通过 `video_url` 访问代理片段。
- 同一 hash 的代理片段不重复上传。
- URL 失效后可以重新上传。

### 3. GLM provider 支持 video_url

当前问题：

```text
GLMProvider 当前只构造 image_url + text。
```

方案一需要支持：

```json
[
  {
    "type": "video_url",
    "video_url": {
      "url": "https://example.com/proxy/sc_0001.mp4"
    }
  },
  {
    "type": "text",
    "text": "请分析这个视频片段..."
  }
]
```

建议改造方向：

```text
VisionProvider
-> analyze_image()
-> analyze_video()
-> analyze_asset()
```

最小实现可以只加：

```python
def analyze_video(self, video_url: str, prompt: str) -> str:
    ...
```

注意：

- 仍然可以走同一个 `chat.completions` 接口。
- 模型建议切到支持 `video_url` 的 GLM-4.6V。
- 当前项目的 `glm-4v-plus` 配置可能不够，应支持模型名配置为 `glm-4.6v`。

验收标准：

- 对一个可访问 mp4 URL 调用 GLM，能返回严格 JSON。
- image 分析能力保留，video 分析能力新增。
- 失败时能落日志并重试。

### 4. Prompt 从“镜头评分”升级为“候选高光判断”

当前 prompt 更像是在问：

```text
这三张图里哪一帧最好，应该截多长。
```

方案一应该问：

```text
这个视频片段是否值得保留？如果值得，推荐保留哪一段？它在故事中适合什么位置？
```

建议输出 JSON：

```json
{
  "keep": true,
  "confidence": 0.82,
  "highlight_score": 8,
  "discard_score": 2,
  "visual_quality": 8,
  "motion_score": 7,
  "summary": "人物从远处走近并展示产品细节",
  "bad_reason": "",
  "recommended_start_offset": 3.2,
  "recommended_end_offset": 12.7,
  "story_role": "hook",
  "tags": ["主体清晰", "动作明确", "适合开头"],
  "reason": "开头动作明显，画面信息密度较高"
}
```

废片输出示例：

```json
{
  "keep": false,
  "confidence": 0.91,
  "highlight_score": 2,
  "discard_score": 9,
  "summary": "镜头长时间对着地面晃动",
  "bad_reason": "画面晃动且无有效主体",
  "recommended_start_offset": 0,
  "recommended_end_offset": 0,
  "story_role": "discard",
  "tags": ["无主体", "晃动", "低信息量"],
  "reason": "该片段不提供叙事或视觉价值"
}
```

验收标准：

- 模型输出能稳定解析。
- 必须有 `keep`、`highlight_score`、`discard_score`、`recommended_start_offset`、`recommended_end_offset`。
- 越界剪点能被本地校正。

### 5. 候选片段池 Candidate Pool

当前项目分析完 scene 后直接进入 storyboard，缺少中间层。

方案一需要新增：

```text
candidate_pool.json
discard_pool.json
```

建议 CandidateClip 字段：

```json
{
  "id": "cand_0001",
  "source": "D:/videos/a.mp4",
  "source_start": 123.2,
  "source_end": 132.7,
  "proxy_scene_id": "sc_0001",
  "summary": "人物展示产品细节",
  "highlight_score": 8,
  "visual_quality": 8,
  "motion_score": 7,
  "story_role": "hook",
  "tags": ["产品展示", "近景"],
  "reason": "主体清晰，动作有信息量"
}
```

分流规则建议：

- `keep=true` 且 `highlight_score >= 7`：进入候选池。
- `keep=true` 且 `highlight_score 5-6`：进入 maybe 池。
- `keep=false` 或 `discard_score >= 8`：进入废片池。
- 每个源视频保留一定比例，避免单个素材垄断最终成片。

验收标准：

- 能看到哪些片段被保留，哪些被丢弃，以及 AI 理由。
- 最终 storyboard 只从候选池选择，不从全部 scene 选择。

### 6. GLM 全局生成 EDL

当前 storyboard 主要靠本地规则排序，方案一需要增加全局 AI 规划。

输入：

```json
{
  "target_duration": 90,
  "user_prompt": "做成有故事感的 90 秒短视频",
  "style": "story_arc",
  "candidates": [...]
}
```

输出：

```json
{
  "target_duration": 90,
  "clips": [
    {
      "candidate_id": "cand_0001",
      "source": "D:/videos/a.mp4",
      "source_start": 123.2,
      "source_end": 132.7,
      "role": "hook",
      "transition": "cut",
      "reason": "画面冲击力强，适合作为开头"
    }
  ]
}
```

规划规则：

- 开头 3-5 秒必须有强画面或强动作。
- 中段按过程或情绪递进。
- 高潮片段放在 60%-80% 位置。
- 结尾需要有收束感。
- 避免重复主体、重复动作、重复构图。
- 不超过目标时长。
- 不选择废片池。

验收标准：

- 能生成完整 EDL。
- EDL 中所有片段都能映射回原始素材。
- 最终时长接近目标时长。
- 多短视频任务中至少能从多个素材来源选择片段。

### 7. render 支持直接执行 EDL

当前 render 消费的是 storyboard 结果。方案一要让 render 能直接执行 EDL。

需要支持：

- source path。
- source_start/source_end。
- 顺序。
- transition。
- target_duration 校验。
- 越界修正。
- 最小时长保护。
- 最大时长保护。

建议：

```text
EDL -> validate_edl() -> render_from_edl()
```

验收标准：

- EDL 能直接渲染为 final.mp4。
- 源文件路径、时间码异常时能明确报错。
- 渲染完成后用 ffprobe 校验成片有效。

### 8. 缓存和成本控制

方案一如果没有缓存，用户成本会很高。

需要缓存：

- 代理片段 hash。
- 上传后的 remote_url。
- GLM 片段分析结果。
- candidate_pool。
- EDL。

缓存策略：

```text
同一素材 + 同一时间窗口 + 同一模型 + 同一分析 prompt
-> 命中缓存，不重复调用多模态模型
```

用户修改剪辑 prompt 时：

```text
不重新分析视频片段
只重新做 EDL 全局规划
```

验收标准：

- 第二次处理同一素材时，多模态调用量明显下降。
- UI 能显示本次预计调用量和实际调用量。

### 9. 前端增加审核体验

方案一不应该只给用户一个最终视频。用户需要快速确认 AI 判断。

需要新增页面或区域：

- 候选高光列表。
- 废片列表。
- 每段缩略图/预览。
- AI 保留理由。
- AI 丢弃理由。
- keep / discard / maybe 状态。
- 重新生成故事弧按钮。
- 最终 EDL 预览。

验收标准：

- 用户不用打开原始 5GB 视频逐段找片。
- 用户能在候选池中快速保留、删除、替换片段。
- 用户能看懂 AI 为什么选这段。

## 模块级改造清单

| 模块 | 当前状态 | 方案一需要补充 |
|---|---|---|
| `frame_extract.py` | 生成三联图 | 保留兜底；新增代理视频片段生成模块 |
| `providers/glm.py` | 发送 `image_url` | 增加 `video_url` 分析 |
| `vision_analyze.py` | 分析单张三联图 | 支持代理视频分析结果解析 |
| `validators.py` | AnalyzedScene | 增加 CandidateClip / EDL schema |
| `storyboard.py` | 本地规则排序 | 新增 AI EDL planner，或让 storyboard 消费 EDL |
| `render.py` | 消费 storyboard | 支持 `render_from_edl()` |
| `backend/app/api/vision.py` | 只代理 chat completions | 可继续透传；另需支持代理文件 URL 或对象存储配置 |
| desktop UI | 创建任务 + 看结果 | 增加候选池审核和 AI 理由展示 |

## 方案一 MVP 推荐开发顺序

### 第一步：打通 video_url 分析

目标：

```text
拿一个代理 mp4 URL，GLM 返回可解析 JSON。
```

任务：

- GLM provider 增加 `analyze_video()`。
- prompt 增加 video candidate 模板。
- 写一个最小测试样例。

### 第二步：生成代理视频片段

目标：

```text
30 分钟视频能被切成 30-60 个低清代理片段。
```

任务：

- 新增 proxy clip 生成。
- 记录 source/start/end/proxy_path。
- 控制片段大小。

### 第三步：上传代理片段

目标：

```text
本地代理片段变成 GLM 可访问的 video_url。
```

任务：

- 对象存储或后端临时 URL。
- 缓存 remote_url。
- URL 失效重传。

### 第四步：候选池

目标：

```text
模型分析结果不直接进 storyboard，而是先进 candidate_pool。
```

任务：

- CandidateClip schema。
- keep/maybe/discard 分流。
- `candidate_pool.json`、`discard_pool.json` 落盘。

### 第五步：AI 生成 EDL

目标：

```text
GLM 根据候选片段生成最终剪辑顺序。
```

任务：

- EDL planner prompt。
- EDL schema 校验。
- 本地兜底排序策略。

### 第六步：按 EDL 渲染

目标：

```text
render.py 可以按 EDL 裁剪原片并生成 final.mp4。
```

任务：

- `render_from_edl()`。
- 时间码校验。
- ffprobe 校验。

### 第七步：前端审核

目标：

```text
用户能看候选片段，不需要自己翻完整素材。
```

任务：

- 候选片段列表。
- AI 理由。
- 保留/删除/重新生成。

## 用户成本控制设计

方案一成本主要来自多模态视频理解调用。

推荐默认配置：

```yaml
cost_control:
  coarse_window_sec: 60
  refine_enabled: true
  refine_top_ratio: 0.3
  cache_enabled: true
  reuse_analysis_on_prompt_change: true
```

长视频成本控制：

- 30 分钟视频先按 60 秒切片，约 30 次多模态调用。
- 只对 top 30% 做精筛，约 9 次额外调用。
- 最终 EDL 规划 1 次文本/多模态调用。

多短视频成本控制：

- 每个短素材先 1 次粗评。
- 低分素材不进入精评。
- 相同 hash 素材跳过。
- 用户改故事风格时只重跑 EDL。

用户界面应显示：

- 预计分析片段数。
- 预计 API 调用次数。
- 已命中缓存数量。
- 本次实际调用数量。
- 失败重试次数。

## 方案一验收标准

长视频：

- 输入 30 分钟以上视频，可以自动生成候选高光池。
- 明显废片能进入废片池。
- 最终成片来自原片裁剪，不是低清代理。
- 用户能查看每个候选高光的 AI 理由。

多短视频：

- 输入多个短素材，可以自动提取候选高质量片段。
- 最终成片不是简单按文件顺序拼接。
- EDL 中能体现 hook、process、climax、ending 等故事角色。
- 重复片段明显减少。

成本：

- 同一素材二次处理能命中缓存。
- 用户修改剪辑 prompt 时不重复做底层视频分析。
- 系统能展示调用量。

稳定性：

- GLM 返回非 JSON 时有重试或修复。
- 推荐剪点越界时能自动 clamp。
- 渲染失败时不会标记成功。
- final.mp4 用 ffprobe 校验有效。

## 最小可交付版本

MVP 不需要一次做完所有能力。建议最小版本如下：

1. 只支持 GLM-4.6V `video_url`。
2. 只支持 30 秒或 60 秒固定窗口代理片段。
3. 只输出 keep/discard/highlight_score/recommended_start/recommended_end。
4. 只做一个候选池。
5. 只做一次 EDL 全局规划。
6. 只支持 cut 拼接，转场和特效后置。
7. 前端只展示候选列表、AI 理由和最终视频。

这版的核心价值是验证：

```text
用户导入素材后，AI 是否能显著减少人工看片和挑片时间。
```

如果这个价值成立，再继续做 ASR、embedding、复杂贴纸、字幕、节奏卡点和素材库。

## 结论

当前项目达到方案一，关键不在继续微调三联图评分，而在增加这条主链路：

```text
低清代理视频片段
-> video_url 多模态理解
-> 候选高光池
-> AI 生成 EDL
-> 本地 FFmpeg 执行
-> 候选审核和成本缓存
```

做到这一步，项目就能从“AI 辅助镜头评分工具”升级为“AI 自动粗剪工具”，基本覆盖视觉型长视频高光提取和多短素材故事弧拼接。
