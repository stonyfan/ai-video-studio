# AI 视频智能剪辑项目商品化开发计划书

## 1. 项目定位

本项目当前阶段已经验证了核心技术可行性：通过 Python 脚本、FFmpeg、场景切分、三联图高光判断、故事弧编排和多平台输出，能够生成质量较满意的短视频成片。

下一阶段目标不是重新验证剪辑效果，而是把现有脚本型原型整理为可销售、可维护、可更新、可扩展的 Windows 桌面端商品。

目标产品形态：

- 用户侧体验：Windows 10/11 桌面应用。
- 实际架构：Windows 桌面客户端内置本地 Python 视频 worker + 云端授权/更新/模型代理服务。
- 核心价值：用户选择本地素材、平台、风格、时长，并可用自然语言提出剪辑要求，系统自动输出适合平台发布的短视频。
- 默认隐私策略：完整原始视频不上传到云端，视频处理和最终渲染在用户本机完成；云端只处理账号、授权、更新、prompt/skill、模型调用代理和必要的轻量分析数据。

## 2. 总体技术路线

### 2.1 推荐架构

```text
Windows 客户端
  - 登录/退出
  - 本地素材选择
  - 任务配置
  - 本地视频处理
  - 查看进度
  - 打开/导出成片
  - 自动更新
  - 内置本地 Python 视频 worker

后端控制服务
  - 账号管理
  - 单点在线控制
  - 使用期限控制
  - 任务管理
  - 版本管理
  - prompt/skill 管理
  - 模型 API 配置管理
  - 模型调用代理
  - 轻量任务记录

本地 Python 视频 worker
  - 素材预处理
  - 场景切分
  - 抽帧/三联图
  - 调用云端模型代理进行 AI 视觉分析
  - storyboard 生成
  - FFmpeg 渲染
  - 日志与结果输出

本地存储层
  - 原始素材引用
  - 中间帧
  - JSON 结果
  - 成片
  - 日志

云端存储层
  - 用户和授权数据
  - prompt/skill/tool 版本包
  - 模型配置
  - 任务摘要
  - 必要时保存压缩后的分析帧或调用记录

后台管理端
  - 用户增删改
  - 密码重置
  - 使用期限配置
  - 任务查看
  - 模型配置
  - prompt/skill 发布
```

### 2.2 部署模式

本项目应优先采用“本地 worker + 云端控制服务”的混合架构，而不是纯 SaaS 云剪辑。

| 模式 | 是否上传完整视频 | 适用场景 | 建议 |
|---|---|---|---|
| 本地剪辑模式 | 否 | 默认商品形态，保护隐私，降低服务器成本 | 第一优先级 |
| 混合分析模式 | 否，只上传抽帧/三联图或 base64 图片 | 使用云端大模型分析画面，但本地渲染 | 推荐主模式 |
| 云端渲染模式 | 是 | 用户电脑性能差、企业批量处理、托管服务 | 后续可选 |
| 私有化部署 | 否或内网传输 | 企业客户、MCN 工作室 | 后续高级版本 |
 
推荐默认链路：

```text
用户选择本地视频
  -> 本地 worker 扫描素材
  -> 本地 worker 抽帧/生成三联图
  -> 客户端/worker 通过后端模型代理上传少量图片
  -> 云端模型返回结构化 JSON
  -> 本地 worker 生成 storyboard
  -> 本地 FFmpeg 渲染成片
  -> 成片保存在用户电脑
```

### 2.3 技术选型建议

| 模块 | 推荐方案 | 理由 |
|---|---|---|
| Windows 客户端 | Tauri 或 Electron | Tauri 安装包小，Electron 开发生态成熟 |
| 后端 API | FastAPI | 负责授权、更新、模型代理和任务摘要 |
| 本地 worker | Python + FFmpeg | 复用现有已验证流程，避免上传完整视频 |
| 后端任务队列 | Redis + RQ/Celery | 仅用于模型代理、云端可选任务和后台异步处理 |
| AI 视觉模型 | Qwen-VL / Doubao / GLM 适配器 | 国内模型可控，后续便于切换 |
| 数据库 | PostgreSQL 或 MySQL | 账号、任务、配置、版本都需要结构化管理 |
| 云端文件存储 | MinIO / OSS/COS | 保存版本包、prompt/skill、少量分析帧；不默认保存完整视频 |
| 本地文件存储 | 用户本机工作目录 | 保存原始素材引用、中间文件和最终成片 |
| 管理后台 | Web 管理端 | 管理账号、配置、任务、版本 |

## 3. 开发阶段计划

### 阶段 0：现有原型整理

目标：把当前实验目录从“脚本现场”整理为可维护项目。

交付物：

- 建立 Git 仓库。
- 分离源码、素材、生成物、模型、临时文件。
- 增加 `.gitignore`。
- 增加 `README.md`。
- 增加 `requirements.txt` 或 `pyproject.toml`。
- 记录当前可复现的 `dao2` 和 `yulan` 成片流程。

完成标准：

- 新环境按文档能安装依赖。
- 能用固定命令跑通至少一个 demo 项目。
- 生成物不再污染源码目录。

### 阶段 1：Python 视频 worker 标准化

目标：把现有多个脚本整理为统一任务入口。

核心输入：

```json
{
  "job_id": "job_001",
  "input_path": "D:/videos/demo",
  "platform": "douyin",
  "style": "fast_cut",
  "target_duration": 30,
  "natural_language_request": "突出调酒过程，节奏快一点，保留成品展示"
}
```

核心输出：

```json
{
  "job_id": "job_001",
  "status": "success",
  "final_video": "outputs/job_001/final.mp4",
  "storyboard": "outputs/job_001/storyboard.json",
  "log": "outputs/job_001/job.log",
  "cost": {
    "vision_calls": 41,
    "estimated_model_cost": 3.8
  }
}
```

交付物：

- `video_worker/` 模块。
- `process_job(config)` 统一入口。
- 标准工作目录结构。
- 进度上报机制。
- 失败日志。
- JSON schema 校验。
- FFmpeg 输出校验。

完成标准：

- 不依赖硬编码本机路径。
- 同一任务可重复执行。
- 失败时能定位到具体阶段。
- 支持素材文件夹输入，并自动过滤非视频文件。

### 阶段 2：AI 模型适配层

目标：让视觉模型、文本模型、prompt 和剪辑策略可配置、可替换。

交付物：

- `model_providers/` 适配层。
- Qwen-VL 适配器。
- Doubao Vision 适配器。
- GLM/百度等预留接口。
- prompt 模板管理。
- 结构化 JSON 输出校验。
- 重试与降级策略。
- 模型调用成本统计。

完成标准：

- 切换模型不影响 worker 主流程。
- 模型返回不合格 JSON 时自动重试。
- prompt 更新不需要改代码。
- API key 不写入客户端。

### 阶段 3：后端控制服务

目标：实现商品化所需的账号、授权、任务摘要、配置、版本管理能力。后端默认不承担完整视频渲染。

核心功能：

- 账号密码登录。
- 单账号只能一处在线。
- 异地登录顶掉旧 session。
- 管理端添加、删除、修改账号。
- 管理端修改密码。
- 管理端配置使用期限。
- 创建本地剪辑任务授权。
- 接收客户端任务摘要和状态心跳。
- 查询任务进度摘要。
- 管理模型 API 配置。
- 管理 prompt/skill 版本。
- 代理模型 API 调用，避免模型 key 暴露到客户端。

完成标准：

- 客户端不直接操作数据库。
- 所有业务操作必须通过后端 API。
- session 失效后客户端必须停止新任务并退出登录。
- 后端能拒绝过期账号创建任务。

### 阶段 4：Windows 客户端 MVP

目标：做出用户可安装、可登录、可在本地提交并完成剪辑任务的桌面端。

用户流程：

1. 输入账号密码登录。
2. 选择多个本地视频，或选择一个文件夹。
3. 文件夹内自动过滤非视频内容。
4. 选择上传平台。
5. 选择视频风格。
6. 选择目标时长。
7. 可选填写自然语言剪辑要求。
8. 点击创建任务。
9. 本地 worker 开始处理素材。
10. 查看本地处理进度。
11. 处理完成后打开或导出成片。

完成标准：

- 支持 Windows 10/11。
- 异地登录后旧客户端能收到退出提示。
- 任务中断后能恢复状态。
- 用户不需要接触命令行。
- 默认不上传完整原始视频。

### 阶段 5：自动更新系统

目标：支持客户端、工具、prompt、skill 的版本更新。

更新类型：

- 客户端程序更新。
- FFmpeg 或工具更新。
- prompt 更新。
- skill 更新。
- 平台模板更新。
- 模型配置更新。

完成标准：

- 客户端启动时检查版本。
- 有更新时提醒用户。
- 用户点击后自动下载并更新。
- prompt/skill 可热更新。
- 程序更新失败时能回滚或保留旧版本。

### 阶段 6：商业化管理后台

目标：让运营人员不依赖开发者即可管理用户和服务。

核心页面：

- 用户列表。
- 新增用户。
- 修改密码。
- 修改使用期限。
- 禁用用户。
- 当前在线状态。
- 任务列表。
- 失败任务详情。
- 模型配置。
- prompt/skill 发布记录。
- 客户端版本发布记录。

完成标准：

- 可完成日常售卖后的账号管理。
- 可追踪用户使用情况。
- 可定位失败任务。

## 4. 核心模块拆分

### 4.1 worker 模块

建议拆分：

```text
video_worker/
  config.py
  job.py
  media_scan.py
  preprocess.py
  scene_detect.py
  frame_extract.py
  vision_analyze.py
  storyboard.py
  render.py
  validators.py
  progress.py
  storage.py
```

开发要求：

- 每个模块只负责一个阶段。
- 模块之间通过明确的数据结构传递结果。
- 不允许跨模块直接读取彼此内部临时文件。
- 不允许业务流程散落在多个一次性脚本中。

### 4.2 API 模块

建议拆分：

```text
backend/
  auth/
  users/
  sessions/
  jobs/
  model_config/
  model_proxy/
  prompt_config/
  updates/
  admin/
```

开发要求：

- 客户端只调用后端 API。
- 客户端本地 worker 只在授权有效时运行。
- 后端不直接控制本地文件路径。
- worker 不直接处理用户账号逻辑。
- 管理后台和客户端共用同一套权限校验。

## 5. 开发约束规则

### 5.1 API 解耦规则

问题风险：API 耦合过高会导致后续新增模型、新增平台、新增风格时大量改动。

规则：

- API 请求和内部任务配置分离。
- 客户端字段不直接等于 worker 内部字段。
- 模型 provider 必须通过适配器调用。
- 平台规则必须配置化，不能写死在渲染代码中。
- prompt 模板必须独立管理，不能散落在业务代码里。
- 后端 API 版本化，例如 `/api/v1/jobs`。
- 外部 API 返回值必须先转换为内部标准结构，再进入主流程。
- 本地 worker 和云端后端通过稳定的 job schema 通信，不能互相依赖内部实现细节。

禁止：

- 在客户端写死模型 API key。
- 在 worker 里直接判断用户权限。
- 在渲染函数里直接读取用户表。
- 在多个模块复制同一套平台规则。
- 默认把完整视频上传到服务器。

### 5.2 测试规则

问题风险：视频处理链路长，任何一环不稳定都会导致系统脆弱。

必须建立四类测试：

1. 单元测试  
   测试路径处理、视频筛选、JSON schema、cut 计算、时长计算。

2. 集成测试  
   测试场景切分、抽帧、三联图、storyboard 生成、FFmpeg 渲染。

3. 端到端测试  
   使用一组小素材，从创建任务到生成成片完整跑通。

4. 回归测试  
   每次修改 prompt、模型适配器、渲染参数后，用固定素材对比输出指标。

最低测试要求：

- 每个核心函数必须有单元测试。
- 每个 worker 阶段必须有可单独运行的测试。
- 每次发布前必须跑通一个 1 分钟以内的小素材端到端任务。
- 所有失败必须输出结构化错误码。

### 5.3 配置管理规则

问题风险：路径、模型、平台参数硬编码会导致无法部署、无法更新。

规则：

- 所有路径来自配置或任务参数。
- 所有模型参数来自配置中心。
- 所有平台模板来自版本化配置。
- 所有 prompt/skill 必须带版本号。
- 每个任务必须记录使用的配置版本。
- 客户端本地 worker 版本、工具版本、prompt/skill 版本必须分开记录。

禁止：

- 代码中出现固定用户路径，例如 `C:\Users\...`。
- 代码中写死某个素材目录。
- 代码中写死某个模型 API key。
- 修改 prompt 后不记录版本。

### 5.4 任务隔离规则

问题风险：本地多个任务并发时，中间文件互相覆盖，或者用户素材被误删。

规则：

- 每个任务必须有独立工作目录。
- 工作目录命名使用 `job_id`。
- 所有中间文件必须写入任务目录。
- 同一用户的不同任务也必须完全隔离。
- 任务完成后按策略清理临时文件。
- 原始素材只读取，不移动、不删除、不覆盖。

推荐目录：

```text
jobs/job_001/
  input/
  work/
  frames/
  triplets/
  json/
  logs/
  output/
```

### 5.5 错误处理规则

问题风险：视频任务耗时长，如果失败信息不清晰，维护成本会非常高。

规则：

- 每个阶段必须有明确状态。
- 每个失败必须有错误码。
- FFmpeg stderr 必须保存到日志。
- 模型调用失败必须记录请求 ID、模型名、重试次数。
- 用户可见错误和开发日志分离。

示例状态：

```text
created
uploading
uploaded
preprocessing
scene_detecting
analyzing
planning
rendering
completed
failed
cancelled
```

### 5.6 安全规则

问题风险：账号、API key、用户素材都属于敏感数据。

规则：

- 密码必须 hash 存储，不允许明文。
- API key 必须加密保存。
- 客户端不保存服务端模型 key。
- 登录 token 必须有过期时间。
- 单账号 session 必须可被服务端强制失效。
- 本地选择文件必须校验类型和大小。
- 默认不得上传完整原始视频。
- 上传分析帧、三联图或日志前必须有明确策略。
- 用户只能访问自己的任务和文件。

### 5.7 更新规则

问题风险：自动更新如果没有版本和回滚机制，会导致用户端不可用。

规则：

- 客户端程序更新和资源更新分开。
- prompt/skill/tool 都必须有 manifest。
- 每个 manifest 必须包含版本号、hash、下载地址、兼容客户端版本。
- 下载后必须校验 hash。
- 更新失败不能破坏当前可用版本。

manifest 示例：

```json
{
  "version": "2026.07.01",
  "type": "prompt_pack",
  "min_client_version": "0.3.0",
  "files": [
    {
      "path": "prompts/food_fast_cut.md",
      "sha256": "..."
    }
  ]
}
```

### 5.8 模型适配规则

问题风险：直接绑定某一个大模型会导致成本、质量和可用性受限。

规则：

- 所有模型走统一接口。
- 模型返回必须经过 schema 校验。
- 支持模型级别重试。
- 支持 provider fallback。
- 每次调用必须记录 token、图片数、耗时、成本估算。
- 商业默认模式下，模型 key 保存在服务端，由后端模型代理调用；客户端只上传必要的抽帧或三联图。

统一接口示例：

```python
class VisionProvider:
    def analyze_scene(self, image_path, prompt, schema):
        ...
```

### 5.9 代码质量规则

规则：

- 不再新增一次性脚本进入主流程。
- 新功能必须进入模块化目录。
- 公共逻辑必须抽成函数或类。
- 禁止复制粘贴大段 FFmpeg 调用逻辑。
- 所有 subprocess 调用必须检查 return code。
- 所有文件读写必须使用 `pathlib.Path`。
- 所有 JSON 输入必须校验字段。
- 所有可变参数必须有默认值和范围检查。

### 5.10 发布规则

每次发布必须包含：

- 版本号。
- 更新内容。
- 数据库迁移说明。
- prompt/skill 版本。
- 客户端兼容版本。
- 已通过的测试清单。
- 已知问题。
- 回滚方式。

## 6. 最小可销售版本范围

MVP 不应该一次性做满所有能力。第一版建议只做以下范围：

- Windows 客户端登录。
- 单点在线控制。
- 本地选择素材或文件夹。
- 平台选择：抖音、小红书、视频号。
- 风格选择：快剪、氛围、自然叙事。
- 时长选择：15s、30s、60s。
- 自然语言补充要求。
- 本地 worker 自动生成成片。
- 客户端显示进度。
- 本地打开或导出结果。
- 管理后台新增/禁用用户、修改期限。

暂缓功能：

- 多语言界面。
- 复杂手动剪辑器。
- 用户自定义高级参数。
- 素材云盘。
- 多人协作。
- 复杂模板市场。

## 7. 风险清单

| 风险 | 影响 | 控制方式 |
|---|---|---|
| 视频文件过大 | 本地处理慢、磁盘占用高 | 本地预估空间、任务恢复、清理策略 |
| 模型输出不稳定 | storyboard 错误 | schema 校验、重试、fallback |
| FFmpeg 参数复杂 | 渲染失败 | 封装统一 render 层、保存日志 |
| API 耦合过高 | 后续难扩展 | provider、platform、style 全部配置化 |
| 测试不足 | 系统脆弱 | 建立小素材端到端测试 |
| 更新失败 | 客户端不可用 | manifest、hash 校验、回滚 |
| API key 泄露 | 成本和安全风险 | key 只在服务端保存，客户端走模型代理 |
| 多任务并发 | 文件冲突、资源抢占 | job 目录隔离、队列限流 |
| 成本不可控 | 商业亏损 | 记录每任务模型调用成本 |
| 用户隐私顾虑 | 阻碍销售 | 默认不上传完整视频，仅上传必要分析帧 |

## 8. 优先级建议

第一优先级：

- worker 标准化。
- 去除硬编码路径。
- JSON schema。
- FFmpeg 输出校验。
- 任务日志。

第二优先级：

- 后端任务 API。
- 登录和 session。
- 管理用户和期限。
- 模型适配层。

第三优先级：

- Windows 客户端。
- 自动更新。
- prompt/skill 版本管理。
- 管理后台完善。

## 9. 阶段验收标准

项目进入商品化测试前，必须满足：

- 一个新账号能登录客户端。
- 同账号异地登录能顶掉旧登录。
- 过期账号不能创建任务。
- 用户能选择本地文件夹并自动过滤视频。
- 一个小素材任务能完整生成成片。
- 失败任务能看到明确错误。
- 管理端能禁用账号。
- prompt 更新不需要重新发客户端安装包。
- 模型 key 不出现在客户端。
- 完整原始视频默认不离开用户电脑。
- 代码仓库不包含大体积素材、模型、虚拟环境。

## 10. 结论

本项目的商品化核心不是引入 Coze 或先制作完整原生剪辑软件，而是先把当前已验证的 Python 脚本型视频处理原型，升级为标准化、可配置、可测试、可调度的 Python 视频 worker。

只有 worker 标准化之后，桌面客户端、后端账号系统、自动更新、模型配置、prompt/skill 管理才有稳定基础。

建议近期开发顺序：

```text
整理项目结构
  -> 标准化 worker
  -> 接入模型适配层
  -> 建后端任务系统
  -> 做 Windows 客户端 MVP
  -> 做管理后台和自动更新
```

该路线能最大程度复用当前已经验证成功的剪辑能力，同时避免过早陷入客户端、平台或工具绑定。
